# TokenRank P0/P1 可信度与增长闭环实施计划

**日期：** 2026-07-19  
**分支：** `codex/p0-p1-product-trust-growth`  
**目标：** 先让公开榜单的每个数字都“说清来源”，再把个人战绩变成可比较、可挑战、可传播的增长入口。

## 1. 现状审计结论

当前首页和移动端榜单已经具备鲜明的竞技视觉，但用户在决定是否安装采集器前仍会遇到三个关键疑问：

1. 榜单数字是否是真实用户数据，还是为了撑场面的 demo 数据。
2. `verified aggregate` 究竟验证了什么，是否等同于 Provider 账单或官方用量。
3. 登录、安装之后除了“看到一个总数”，还能获得什么持续价值和分享理由。

本次实施不改变现有 Token 计分机制，也不把 Token 用量包装成能力、效率或产出质量。永久产品承诺改为“AI 活动有记录、可比较”，原 `BURN TOKENS. ASCEND RANKS.` 只保留为赛季或活动口号，不再承担长期价值主张。

## 2. P0：可信度与首次转化

| 项目 | 实施内容 | 验收标准 |
| --- | --- | --- |
| Demo 隔离 | 公共榜单、公开个人页、公开 sitemap 默认排除 `demo_` 用户；仅允许非生产环境显式开启；种子脚本增加显式许可和生产保护 | 生产环境无法展示内置 demo 用户；真实账号不受影响；有单元测试覆盖 |
| 永久定位 | 首页改为 `YOUR AI ACTIVITY. RANKED.` / `你的 AI 活动，公开排名`，同时明确只上传聚合用量 | 中英文 Hero、metadata、OG、README、`llms.txt` 口径一致 |
| 信任说明 | 榜单附近展示“X 身份已连接 / 本地聚合 / 服务端校验”三层含义，并明确“非 Provider Verified、非生产力评分、金额为估算” | 页面不再出现含混的 `verified aggregate`；规则页与 FAQ 可直接回答数据来源问题 |
| 登录前价值 | 未登录用户先看到无需账号的 `npx --yes tokenrank preview`，可以在上传前检查本机聚合结果，再决定是否绑定身份 | `/onboard` 未登录状态可复制预览命令；明确预览不会上传数据 |

## 3. P1：个人战绩与挑战分享

| 项目 | 实施内容 | 验收标准 |
| --- | --- | --- |
| 排名上下文 | 公开个人页增加 7 天名次、参赛人数与 Top 百分比 | 有 7 天数据时显示真实排名；无数据时明确显示未进入本期榜单 |
| 活跃节奏 | 计算当前连续活跃天数、最近 7 天 Token 与前 7 天变化 | UTC 边界和无历史基线有单元测试；不显示无穷百分比 |
| 分享文案 | X 分享文案包含 7 天名次、连续活跃和变化信息 | 有竞争上下文时分享文案不再只有累计总量 |
| 挑战链路 | 个人页提供“挑战我的排名”，进入 `/onboard?challenge=<handle>`；上榜页展示挑战来源 | 链接可访问、参数经过规范化；无效或非公开账号不会伪造挑战信息 |
| 移动端信息层级 | 竞争卡、信任标签和 CTA 在窄屏按单列排列，避免横向溢出 | 真实移动视口完成首页、上榜页和个人页视觉验收 |

## 4. 本轮不纳入

- GitHub 登录与多身份模型：现有公开路由以 X handle 为主键，需要单独设计 participant identity 与迁移，不能把 GitHub 用户名静默写进 `xHandle`。
- Provider/Admin 官方账单连接与 `Provider Verified` 榜：属于 Evidence V2 的后续能力；本轮只把现有信任边界说准确。
- 永久 `Founding 100` 身份、赛季系统、邀请归因落库：需要不可变资格规则、数据库迁移与漏斗事件模型，本轮先交付可传播的挑战链接。
- CLI 安装器和扫描逻辑改造：CLI 已支持登录前 `preview`；本轮只在 Web 转化路径中暴露它，不修改相邻仓库。
